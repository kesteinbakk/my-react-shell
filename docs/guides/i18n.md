# i18n module

The `t()` seam: a zero-dependency translation provider + hook, a central message
catalog contract, and a dev-only missing-key surface. Self-contained, Convex-free,
and router-free — shipped at the sub-path **`my-react-shell/i18n`** so the package
barrel stays the theme core.

```ts
import {
  I18nProvider,
  useTranslation,
  MissingTranslationsOverlay,
  translateNow,
  mergeMessages,
} from 'my-react-shell/i18n'
```

## What it does

- A single `<I18nProvider messages={…}>` owns the active locale, persists it to
  localStorage, and detects an initial locale from the browser.
- `useTranslation(namespace?)` returns `{ t, locale, locales, setLocale }`.
- `t(key, params)` resolves a dotted key against the active locale's catalog,
  interpolates `{{param}}` placeholders, and falls back to the fallback locale,
  then to returning the key itself.
- Every unresolved key is recorded (in dev) and surfaced by
  `<MissingTranslationsOverlay>` and a console warning — so missing strings are
  visible, never silent.

It does **not** bundle any catalogs — the **consumer owns all message data**. The
module ships the seam, the translator, and the dev surface.

## Wire it

```tsx
import { I18nProvider, useTranslation } from 'my-react-shell/i18n'

const messages = {
  en: { nav: { home: 'Home' }, greeting: 'Hello {{name}}' },
  no: { nav: { home: 'Hjem' }, greeting: 'Hei {{name}}' },
}

function App() {
  return (
    <I18nProvider
      messages={messages}
      locales={[{ code: 'en', label: 'English' }, { code: 'no', label: 'Norsk' }]}
      defaultLocale="en"
    >
      <Page />
    </I18nProvider>
  )
}

function Page() {
  const { t, locale, locales, setLocale } = useTranslation()
  return (
    <>
      <h1>{t('nav.home')}</h1>
      <p>{t('greeting', { name: 'Kari' })}</p>
      <select value={locale} onChange={(e) => setLocale(e.target.value)}>
        {locales.map((l) => <option key={l.code} value={l.code}>{l.label}</option>)}
      </select>
    </>
  )
}
```

`<I18nProvider>` props: `messages` (required — `Record<locale, nested catalog>`),
`locales`, `defaultLocale` (must be a `messages` key — throws otherwise, no silent
default), `fallbackLocale`, `detectBrowserLocale` (default `true`), `storageKey`,
`resolve`, `onMissingKey`, `debug`.

A `namespace` scopes keys: `const { t } = useTranslation('nav')` lets you call
`t('home')` instead of `t('nav.home')`.

## Central-key policy

Every user-facing string goes through `t()` against the **central catalog** — no
inline literals in components, no per-component string maps. Namespace keys by
feature (`account.*`, `nav.*`, `errors.*`) so a key's owner is obvious and the
catalog stays navigable. Compose a module's base strings under a consumer's catalog
with `mergeMessages(base, override)` (deep merge, override wins) rather than copying
keys.

## Missing-key surface (dev)

Mount once near the app root:

```tsx
import { MissingTranslationsOverlay } from 'my-react-shell/i18n'
// …
<MissingTranslationsOverlay />
```

It renders nothing in production (gated on `import.meta.env.DEV`) and nothing when
no key is missing. While keys are missing it shows a fixed, warning-styled panel
listing each `key · locale`, with a **Copy keys** action (writes the distinct missing
keys, one per line, to the clipboard, paste-ready for the catalog) and a **Clear**
action. The same data is available programmatically via
`missingKeyStore` (`subscribe` / `getSnapshot` / `clear`), and every miss also logs a
console warning and calls the provider's `onMissingKey` if supplied.

## Imperative translation

For callers outside React render (event handlers, toast/error helpers) use
`translateNow(key, params)` — it reads the mounted provider's translator. Inside
components prefer `useTranslation().t`, which re-renders on locale change.
`translateNow` assumes a single active provider (the usual app-wide case).

## Typed keys (opt-in, compile-time)

By default the seam is **stringly-typed**: `t('any.key')` accepts any string, so a
typo or a key missing from the catalog surfaces only at runtime (the missing-key
surface above). To catch it at **compile time**, bind your key union once with
`createTypedI18n` and import the typed hooks it returns instead of the bare ones:

```ts
// src/i18n.ts — declare the typed seam once
import { createTypedI18n } from 'my-react-shell/i18n'
import type { DotPaths } from 'my-react-shell/i18n'
import { en } from './catalog' // your nested catalog object for the default locale

// Derive the key union from the catalog…
export type Key = DotPaths<typeof en> // 'nav.home' | 'greeting' | …
// …or hand-write it: type Key = 'nav.home' | 'greeting'

export const { useTranslation, useT, translateNow } = createTypedI18n<Key>()
```

```tsx
import { useTranslation, useT, translateNow } from '@/i18n' // typed, not 'my-react-shell/i18n'

function Account() {
  const { t } = useTranslation()
  t('nav.home') // ✅ ok
  t('nav.nope') // ❌ compile error — not a key

  const tt = useT() // no-namespace convenience: the bound `t` directly
  return <h1>{tt('greeting')}</h1>
}

translateNow('nav.home') // ✅ imperative callers are typed too
```

- **`useT()`** is the no-namespace convenience — it returns the bound `t` directly,
  matching the ergonomics apps coming from the SolidJS foundation expect.
- **Namespaced narrowing** works on the factory's hook: `useTranslation('nav')`
  narrows `t` to the sub-keys of your union under `nav.`, so `t('home')` is checked
  against `nav.home`. *(The bare exported `useTranslation<K>('nav')` keeps the full
  `K` — narrowing the union and the namespace together per call needs partial
  type-argument inference TypeScript doesn't offer, so use the factory for
  namespaced typing.)*
- **Non-breaking.** Every generic defaults to `string`, and `createTypedI18n` is pure
  typing sugar over the same hooks — no new provider, context, or runtime. The untyped
  `useTranslation()` / `translateNow()` are unchanged; adopt typed keys in one file and
  leave the rest stringly-typed.
- **Types, not parity.** Typed keys catch typos and unknown keys. They don't replace
  locale **parity** — the union comes from one catalog, so a key present in one locale
  but missing in another is still a runtime miss caught by the missing-key surface.

The contract types are generic too — `TFunction<K>`, `I18nContextValue<K>`,
`UseTranslationResult<K>`, and `NamespacedKeys<K, NS>` — all defaulting to `string`.

## Bring your own

Two paths, by how much you want to replace:

- **Keep the seam, swap the engine.** Pass `resolve={(locale, key, params) => …}`
  to delegate lookup to ICU / react-i18next / your own runtime while keeping this
  provider's locale state, persistence, and missing-key surface. Returning
  `undefined` triggers the normal missing-key handling.
- **Replace it wholesale.** Don't use `<I18nProvider>`; satisfy the exported
  contract (`I18nContextValue`, `TFunction`, `Messages`, `Locale`) so your app code
  is shaped the same and could move back to the default later.
