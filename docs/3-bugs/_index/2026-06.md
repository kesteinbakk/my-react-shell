| B003 | nested-popper-dismisses-dialog | awaiting-user-confirmation | — | **Filed 2026-06-30 (reported from offansk-ev Invite-member modal).** A `Select` (or any Radix popper) **inside** a `Dialog`/`Sheet`/`ConfirmDialog`: open the popper, then click inside the surface but outside the popper, and the **whole surface closes** — losing the user's in-progress data. Ruled out the `link:` Radix-dedup hazard by instrumentation (unified single Radix context). Root cause: while the popper owns the layer Radix forces the dialog content to `pointer-events: none`; the click falls through, Radix closes the popper, then **re-evaluates the same gesture** as an outside-interaction for the dialog and dismisses it — fired **deferred**, *after* the popper unmounts, so a "is a popper open now?" check is too late. **Fix (shell-side):** `src/components/useDialogDismissGuard.ts` — a shared hook that records (via a capture-phase `pointerdown` listener) whether a popper was open at **gesture start**, and `preventDefault()`s the surface's `onPointerDownOutside`/`onInteractOutside` for that gesture. Wired into `Dialog`, `Sheet`, `ConfirmDialog`. Verified in `my-react-shell-demo` (bug gesture survives; option-select / backdrop / Esc all still correct). | `src/components/useDialogDismissGuard.ts` (new), `src/components/{Dialog,Sheet,ConfirmDialog}.tsx`; demo `OverlaysDialogsPage.tsx`. |
| B002 | pageheader-unstable-thunk-loop | open | — | **Filed 2026-06-20 (dogfood from evaluering T029).** `<ShellPageHeader>` **infinite-loops** ("Maximum update depth exceeded", crashes to the consumer's error boundary) when a consumer passes an **unstable thunk** for `tabs` / `title` / `actions` — e.g. the idiomatic `tabs={() => <PageTabs … />}`, a fresh function each render. The registration effect (`ShellPageHeader.tsx:54-72`) lists `props.tabs` (et al.) in its deps; an identity change re-runs the effect, which calls `registerPageHeader(spec)` → `setState(pageHeaderSpec)` → AppShell re-renders → consumer re-renders → new thunk → re-register → loop. The docstring (`ShellPageHeader.tsx:8-11`) explicitly tells consumers stable thunks are optional ("correctness does not depend on it: re-register is idempotent") — that is **false**; an unstable thunk is fatal. **Fix (shell-side):** make registration idempotent against an equivalent spec (skip the `setState` when the new spec is shallow-equal to the current), OR drop the thunk props from the effect deps and read them via a ref, OR change the doc to mandate stable thunks (`useCallback`) as a hard requirement (and ideally warn in DEV when a thunk identity churns). Evaluering worked around it consumer-side (memoizing the thunk) but the footgun should not exist. **Related:** the same project also needed a `link:`-consumer **Radix dedup** — the shell resolves its `@radix-ui/*` peers from its own `node_modules`, so a `link:` consumer gets two physical Radix copies and `composeRefs` loops in the breadcrumb dropdown; consumers must `resolve.dedupe` the Radix graph (like React). Worth a line in the distribution-model guide. | `src/app-shell/ShellPageHeader.tsx` (registration effect + docstring); `docs/guides/distribution-model.md` (Radix dedup note). |
